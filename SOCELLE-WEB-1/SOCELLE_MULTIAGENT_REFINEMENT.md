## A3 — Rules for Acceptable Refinement

| RULE | DETAIL |
| --- | --- |
| Shorter is always better | Premium brands do not over-explain. If a paragraph can become two sentences without losing meaning, make it two sentences. If a sentence can become a phrase, make it a phrase. |
| Preserve the strategic claim | Every refinement must preserve the original competitive or strategic claim. You may change how it is said. You may not change what is being claimed. |
| Specificity over abstraction | Replace vague language with specific language. "Better insights" becomes "purchasing benchmarks by category." "Great brands" becomes "verified professional brands." Specificity is a premium signal. |
| No SaaS language | SOCELLE is infrastructure for professional beauty, not a tech product for general business. Reject: streamline, leverage, unlock, empower, scalable, solutions, ecosystem. Accept: platform, marketplace, commerce layer, relationship, account, verified. |
| Premium tone test | Read the refined copy and ask: would this appear on a Four Seasons website or a Mailchimp website? Premium tone is restrained, specific, and assumes the reader is intelligent. It does not exclaim, urgency-bait, or over-promise. |
| Gate before deploy | No hero copy, no section headline, and no CTA language ships without approval at the relevant gate (G2 for Homepage, G3 for For Buyers, G4 for IH architecture). Agent 6 submits; owner approves. |

## A4 — Rules for What Must Not Change

| RULE | DETAIL |
| --- | --- |
| Do not change the economics structure | Commission-only with no monthly fees is not a marketing claim. It is a structural product fact that operators and brands use to evaluate risk. Any language that implies upfront cost or recurring fees is wrong and damaging. |
| Do not neutralize the vertical claim | Professional beauty is the market. Do not generalize to "wellness" or "professional services" in hero-level copy. Vertical specificity is the positioning strategy. Broader language dilutes it. |
| Do not remove the verification framing | Licensed buyers and reviewed brands are the moat. Copy that softens the curation language ("anyone can join," "open to all") destroys the value proposition for both sides of the marketplace. |
| Do not rewrite the About page argument | The four-problem structure is the intellectual core of the brand. Summarizing it ("the supply chain is broken") loses the argument. The argument is the trust engine. Leave it intact. |
| Do not simplify the two-sided structure | Both portals (brands and buyers) must appear as full, distinct value propositions throughout the site. Any copy that collapses this into a single audience or a single CTA misrepresents the platform. |

# B — Intelligence Hub Hook Refinement
The Intelligence Hub is the most strategically important differentiator in the platform — but only if it can be described publicly without sounding like a dashboard, a blog, or a generic analytics product. The current internal thesis is sophisticated but not yet translatable to a single-sentence acquisition hook. This section does that translation.
## B1 — What the IH Is Not (Public Framing Guard Rails)

| Not a dashboard | Do not describe it as a dashboard, tool, or product feature. Dashboards are operational. The IH is strategic intelligence. |
| --- | --- |
| Not an analytics product | Do not use analytics, data visualization, tracking, or metrics language. This positions it as a reporting tool rather than an intelligence layer. |
| Not a blog or content library | Do not describe it as resources, articles, guides, or education hub in the primary hook. These are secondary frames that undersell the commercial value. |
| Not a feature list | Do not lead with a bullet list of capabilities. Lead with the outcome the operator experiences. The specific features (benchmarks, purchasing data, education) are revealed one layer deeper. |
| Not exclusive to tech-forward operators | The hook must work for a medspa owner in Nashville who has never used a purchasing benchmark in her life. Jargon-free, benefit-forward, specific. |
| Not gated from the first sentence | The public-facing hook must make operators want to create an account. It should not mention "gated" or "account required" until the CTA level. |

# C — Launch Scope Compression
The current coordination document defines six priority pages (P1–P6) as the immediate launch set. This is the right set for a two-week sprint with no external constraints. In practice, the fastest path to a credible public launch — and the fastest path to operator acquisition — requires a three-page first cohort. Everything else follows within days.
## C1 — True First-Launch Page Set
These three pages form a complete, standalone acquisition loop for operators. A visitor from any referral source can arrive, understand the platform, and convert. The loop is: Homepage → For Buyers → Request Access.

| PAGE | FIRST-LAUNCH RATIONALE |
| --- | --- |
| P1 — Homepage | Primary entry point for cold traffic. Sets visual standard for all pages. Must ship first. Blocks no other page but defines the visual and copy standard every other agent follows. Gate G2 approval required before any other Wave 3 page launches. |
| P2 — For Buyers / Operators | Primary conversion page for the operator acquisition cohort. Spa directors, medspa owners, salon operators are the first audience. This page is the reason to build the Intelligence Hub. It should be the best page on the site at launch. |
| P3 — Request Access | The conversion endpoint for both audiences. Not a form. A conversion page with a form. Captures intent from operators and brands. Without this page, P1 and P2 have no landing point for CTA clicks. This ships in Wave 3B, parallel to P1 and P2. |

These pages should ship within 3–5 days of P1+P2+P3. They are not delayed — they are sequenced. Their copy is fully written and their visual tokens are already resolved by VISUAL_RESET. Agent time on these pages is low.

## C4 — Risks of Overbuilding Too Early
- More pages means longer QA before any page is live. Every additional page in Wave 3 extends the pre-launch QA cycle and delays P1.
- The IH teaser landing page, if it ships before operator accounts exist, describes a product the visitor cannot yet access. This is a credibility problem, not a feature gap.
- Spreading agent attention across six parallel pages reduces the visual quality ceiling of the three pages that matter most. P1 and P2 deserve the best agent work on the site. They cannot receive it if six pages are being built simultaneously.
- Premature IH landing page creates platform support obligations. If operators find the page and attempt to access the IH before it exists, the request pipeline will surface a gap the team cannot close until post-launch Wave 6.
- A three-page soft launch with a tight operator acquisition loop is a better story than a six-page launch with diluted quality. The story is: "We launched three pages and operators are signing up." That story funds the next build wave.
