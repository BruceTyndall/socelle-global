## 1. PLATFORM IDENTITY

SOCELLE is a living beauty and wellness intelligence portal.

**It is NOT:**
- a generic marketplace
- a static directory
- a blog
- a dashboard product with flat card grids
- a brochure site
- a beauty brand

**It IS:**
- intelligence-first platform (marketplace second)
- operator/provider/brand acquisition engine
- retention engine powered by education, news, events, and live intelligence
- public brand-profile ecosystem (500+ auto-generated, claimable)
- jobs marketplace and labor-intelligence layer
- premium affiliate commerce layer (day 1)
- 90% automated wherever lawful and feasible
- cross-vertical across spa, medspa, skincare, hair, salon, makeup, wellness, holistic health, cosmetic derm, cosmetic/plastic surgery, devices, beauty tech, and beauty AI
- operating at local, regional, national, and international levels

**Revenue streams (7):**
0. Premium affiliate commerce (day 1)
1. Embedded poll sponsorship ($1K–$5K)
2. Brand intelligence reports ($500–$5K)
3. Sponsored quizzes/studies ($5K–$25K)
4. Brand claim subscriptions ($199–$999/mo)
5. Premium operator subscriptions ($49–$149/mo)
6. Jobs marketplace (employer posting fees + featured listings)

**Tech stack:** Vite + React 18 + TypeScript + Tailwind CSS + Supabase (PostgreSQL + Edge Functions + pg_cron + pgmq + Realtime + Storage). Deployed on Cloudflare. Supabase-first architecture — no JSON files as primary data store.

## 4. DUAL-REQUIREMENT DOCTRINE

**This rule is non-negotiable:**

SOCELLE only wins if it satisfies BOTH at the same time:
1. **Maximum lawful intelligence density**
2. **High-end page flow / visual artistry**

Agents must never optimize one without the other.

### 4.1 Intelligence density rules
- Harvest as much lawful beauty and wellness data as possible
- Use free/public sources first
- Combine weak signals into stronger derived intelligence
- Confidence-score computed outputs (0.00–1.00)
- Show freshness wherever possible
- Prioritize commercially useful intelligence, not vanity metrics
- No fake data, no hardcoded "live" claims in production

### 4.2 Page artistry rules
- Page flow is product behavior, not decoration
- Composition, rhythm, hierarchy, and conversion flow matter as much as the data
- One strong signal beats six weak ones

- No flat dashboard grids as default layout
- No brochure-site sections with no product tension
- No blog-with-widgets behavior

## 17. ARCHITECTURE — SUPABASE-FIRST

| Pattern | Implementation |
|---|---|
| Primary database | Supabase PostgreSQL (NOT JSON files) |
| Ingestion | pg_cron → pg_net → Edge Functions → pgmq queues |
| Intelligence | Materialized views, refreshed CONCURRENTLY every 15 min |
| Serving | PostgREST (auto CRUD) + Edge Functions (custom logic) |
| Live updates | Realtime Broadcast (not Postgres Changes — for 500+ brand scale) |
| Storage | Supabase Storage + Smart CDN for cached artifacts |
| Auth | Supabase Auth with role-based RLS |
| Access gating | account_plans table gates MV/endpoint access by plan tier |
| Rate limiting | PostgreSQL-based atomic counter per service |
| Search | Generated tsvector columns with GIN indexes |

| RSS in-memory parse | Edge Function + rss_items table + full-text search |
